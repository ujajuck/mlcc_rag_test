# MLCC RAG 스펙 셀렉터

고객 MLCC 요구사항을 카탈로그 기반 후보로 변환한다. 코드 매핑은 **catalog-codebook.md에서 직접 해석**하고, 패밀리/앵커 등 맥락 정보만 `search_rag`로 보충하며, 필요 시 활성 라인업 DB 조회를 수행한다. 이 스킬은 **사전 선정과 근거 기반 추론**을 위한 것이며, 최종 주문 가능 P/N 확정 용도가 아니다.

## 실행 원칙

- tool 호출이 실패하거나 결과가 0건이어도 즉시 포기하지 않는다. 조건을 완화하거나, 다른 파라미터로 재시도하거나, 사용자에게 대안을 제시한다.
- 사용자가 "다시 해봐", "다른 방법으로", "범위 바꿔서" 등 재시도를 요청하면 반드시 응한다. 이미 시도한 조건을 변경하여 최소 2~3회는 다른 접근을 시도한다.
- "안됩니다"로 끝내지 말고, 항상 다음에 시도할 수 있는 대안(조건 완화, 와일드카드 확장, 다른 축 변경 등)을 함께 제시한다.

## Reference 문서

각 단계를 실행할 때 해당 reference를 반드시 참고한다.

- `references/catalog-codebook.md` **(최우선, 반드시 가장 먼저 참고)**: 코드 매핑, 패밀리 라우팅, 청크 조회 타깃, 사이즈/두께 필터링 규칙, 조건 완화 맵, 카탈로그 가드레일. 스펙 해석의 기준이 되는 코드북이므로 search_rag 호출 전에 반드시 이 문서의 코드 매핑을 먼저 확인한다.
- `references/search-playbook.md`: 검색 순서, 쿼리 패턴, 랭킹, 응답 포맷. **search_rag 호출 전에 반드시 참고한다.**
- `references/active-lineup-lookup.md`: 부분/모호 요청, `chip_prod_id` 기준 현재 유통 제품 확인, `search_query_database`를 통한 인접기종 탐색. **활성 라인업 조회나 인접기종 탐색 전에 반드시 참고한다.**
- `references/prompt-examples.md`: 한국어/혼합 언어 호출 예시

## 금지 규칙
1. **search_rag 호출 시 반드시 search_group 필터를 지정하라.** 필터 없는 넓은 검색은 관련 없는 청크가 섞이므로 금지한다.
   - 허용 필터: `"family_reference"`, `"caution_reference"`, `"dimension_reference"`
   - 금지 필터: `"mapping_core"`, `"mapping_support"` (codebook으로 해결)

2. **한 턴에서 search_rag를 3회 초과 호출하지 마라.**
   - 일반 선정: 최대 2회 (패밀리 1 + 앵커 1)
   - 주의 특성 포함: 최대 3회

3. **다중 조건 스펙의 코드 매핑은 codebook에서 한번에 해석한다.** 조건이 5개든 7개든 codebook 한번 읽으면 모든 position을 동시에 확정할 수 있다. 하나씩 search_rag로 검색하면 안 된다.

### 잘못된 예 (하지 마라)

- `search_rag("X5R temperature code")` → 금지. codebook에서 `A = X5R` 직접 확인
- `search_rag("0201 0603 X5R 4.7uF 4V M tolerance")` → 금지. 다중 조건 한번에 RAG 검색은 노이즈 발생
- `search_rag("size code 03")` → 금지. codebook에서 `03 = 0201/0603` 직접 확인
- `search_rag("voltage code R 4V")` → 금지. codebook에서 `R = 4.0Vdc` 직접 확인

### 올바른 예 (이렇게 하라)

- codebook에서 `A=X5R, R=4V, M=±20%, 03=0201/0603` 모두 확정 → search_rag **0회**
- `search_rag("high level II outdoor 85 85", search_group="family_reference")` → 패밀리 맥락 1회
- `search_rag("CL03 X5R 4.7uF 4V", search_group="dimension_reference")` → 앵커 탐색 1회

## 운영 경계

- 온도특성 코드, 전압 코드, 용량 코드 규칙, 편차 코드, 사이즈 코드, 두께 코드 필터, 신뢰성 패밀리 설명, 조회된 청크 내 명시 예시 P/N은 **카탈로그를 기준 데이터**로 간주한다.
- 활성 라인업 DB 조회는 `mdh_continous_view_3` 뷰에서 **일치하는 `chip_prod_id` 행의 현재 존재 여부만** 확정 정보로 간주한다.
- DC 바이어스 거동, 고주파 유효 용량, ESR/ESL 정확값, 정확한 주문 꼬리코드, 실제 라인업 가용성은 대상 파트에 대해 청크에 직접 명시된 경우를 제외하면 **검증 항목**으로 취급한다.
- 주의 그래프/특성 곡선은 샘플 가이드이며, 후보 파트의 정량 보증치로 환산하지 않는다.
- 8~11번째 코드가 근거 청크로 직접 뒷받침되지 않으면 완전한 part number를 임의 생성하지 않는다. 스켈레톤 또는 TBD로 표기한다.

## 워크플로우

> **표기**: [필수] = 항상 실행, [조건부] = 조건이 맞을 때만 실행

0. [필수] **가장 먼저** `catalog-codebook.md`를 읽어서 모든 position(1-7) 코드를 확정한다. 이 단계에서 search_rag를 호출하면 안 된다. 다중 조건이어도 codebook만으로 완결된다.

1. [필수] 사용자 요청을 다음으로 분해한다.
   - 하드 제약: 온도특성 코드, 정격전압, L/W/T 최대치, 편차, 명시된 패밀리/신뢰성 요구
   - 소프트 제약: 적용처 힌트, 선호 기준용량, 패키징 선호, 앵커 근접성
   - 검증 전용 제약: 바이어스/주파수 조건 유효용량, ESR/ESL, 리플, 정확 꼬리코드
   - 활성 라인업 조회 트리거: 부분 요청, 모호 요청, 현재 유통 여부 질문, 카탈로그 추론 후에도 미해결 코드 위치 존재
   - 인접기종 탐색 트리거: 인접/contiguous 모델 요청 시 `search_query_database`로 `public.mdh_contiguous_condition_view_dsgnagent` 대상 SQL 조회
2. [필수] 검색 전 단위를 정규화한다.
   - 길이 → `mm`
   - 용량 → `uF` 및 E-series 코드 후보
   - 전압 → `Vdc`
   - 편차 표현(`M` 등) → 코드/퍼센트 병행 해석
3. [필수] `catalog-codebook.md`의 핵심 코드 맵에서 온도특성, 전압, 용량, 편차, 사이즈, 두께 코드를 직접 확정한다. 모든 position(1-7)을 한번에 해석하며, search_rag를 호출하지 않는다.
4. [조건부] 적용처 힌트가 있으면 `search_rag`를 **1회** 호출해 패밀리/신뢰성 근거를 조회한다(`search_group="family_reference"`). 힌트가 없으면 Standard로 가정하고 생략한다.
5. [조건부] 앵커 파트가 필요하면 `search_rag`를 **1회** 호출해 확정된 코드 조합(사이즈+온도특성+전압+용량)으로 근접 예시를 탐색한다. 자유 추정보다 카탈로그 인접 예시를 우선한다.
6. [필수] 카탈로그 근거로 하나 이상의 `후보 스켈레톤`을 구성한다. 요청이 불완전하면 알려진 코드 위치를 유지하고 미해결 1문자 슬롯을 `_`로 둔 `chip_prod_id 조회 패턴`도 함께 만든다.
7. [조건부] 활성 라인업 조회 도구가 있고 요청이 부분적/모호하거나 현재 유통 확인이 필요하면, 최종 답을 억지로 확정하기 전에 해당 패턴으로 조회한다.
   - 인접기종 검색은 `search_query_database`를 사용하며, `public.mdh_contiguous_condition_view_dsgnagent`에 대해 `ILIKE` 기반 SQL SELECT를 작성한다.
8. [조건부, 7단계 수행 시] DB 조회 결과가 다건이면 목록을 제시하고, 미해결 슬롯/요구사항을 좁히는 **단일 후속 질문**을 한다. 임의 선택하지 않는다.
9. [조건부, 8단계 결과 0건] DB 조회 결과가 0건이면 **조건 완화 루프**를 수행한다.
   a. 현재 스켈레톤에 인코딩된 스펙 축을 식별한다. 예: `CL32_106_O____`는 사이즈(`32`=1210), 용량(`106`=10uF), 전압(`O`=16V)을 고정한다.
   b. 고정된 축을 사용자에게 보여주고, 어떤 축을 상/하/양방향으로 완화할지 묻는다.
      - 사이즈: `사이즈를 한 단계 키워 1812(43)로 검색할까요, 또는 줄여 0805(21)로 검색할까요?`
      - 용량: `용량을 4.7uF(475)로 낮추거나, 22uF(226)로 올려 검색할까요?`
      - 전압: `전압을 25V(A)로 올리거나, 10V(P)로 낮춰 검색할까요?`
   c. 사용자가 변경 축을 선택하면 새 코드로 스켈레톤을 재구성하고 `active_lineup_lookup`을 재실행한다.
   d. 재시도도 0건이면 (b)로 돌아가 추가 완화/축 변경을 제안한다. 이미 2개 이상 축을 완화했는데도 실패하면 `%` 와일드카드 확대 또는 `_` 확장을 제안한다.
   e. 히트가 나오면 8단계(다건 축소) 또는 10단계(종합)로 진행한다.
10. [조건부] 사용자가 바이어스/주파수/임피던스/에이징 거동을 요청할 때만 `caution_characteristics`를 조회하고, 데이터시트/실측 검증 필요 항목을 명확히 남긴다.
11. [필수] 최종 응답은 아래 구조로 합성한다 (응답 계약 참조).
   - 제약조건 요약
   - 스켈레톤 조립 과정 (형식 A 트리 다이어그램 또는 형식 B 번호 주석)
   - (조회 시) DB 활성 라인업 히트
   - 검증 필요 항목

## 검색 규칙

> 검색 순서, 쿼리 패턴, 컬렉션 라우팅, 호출 예산 상세는 `references/search-playbook.md`를 참고한다.

- positions 1-7 코드 매핑은 `catalog-codebook.md`에서 직접 확정한다. search_rag로 코드 테이블을 검색하지 않는다.
- search_rag는 codebook에 없는 맥락 정보(패밀리/신뢰성/앵커/주의 특성)에만 사용한다. 호출 시 `search_group` 필터 필수.
- `collection="context"` (기본값)를 권장한다. `collection="core"`는 일반 경로에서 사용하지 않는다.
- 미해결 코드 위치가 있으면 스켈레톤을 DB 조회 패턴(`_` 슬롯)으로 변환하고 강제 추정보다 대화형 축소를 우선한다.
- DB 조회를 나중으로 미루지 않는다. 패턴이 준비되면 해당 턴에서 즉시 실행한다.

## 응답 계약

### 인접기종 검색 시 (간결 형식)

인접기종/contiguous/adjacent model 요청이면 아래 3단 구조로 간결하게 응답한다. 자세한 형식과 예시는 `references/active-lineup-lookup.md`의 "인접기종 검색 시 응답 형식" 섹션을 참고한다.

1. **스켈레톤 조립 과정** — 요청 스펙에서 각 코드 위치가 어떻게 결정되었는지 서술
2. **DB 조회 및 결과** — 조립된 패턴을 tool에 입력하고 결과 목록 제시
3. **간단한 후속 안내** — 다건이면 축소 질문, 0건이면 완화 제안 (한두 문장)

7단 전체 응답 구조는 사용하지 않는다.

### 일반 스펙 선정 시 (전체 형식)

일반 스펙 선정 요청이면 아래 순서를 유지한다.

1. **제약조건 요약** — hard/soft/검증 전용 분류
2. **스켈레톤 조립 과정** — codebook 기반 코드 매핑 + 후보 스켈레톤 제시. `references/prompt-examples.md`의 **형식 A(트리 다이어그램)** 또는 **형식 B(번호 주석)** 중 하나를 사용해 각 코드 위치의 의미를 시각적으로 보여준다.
3. **(조회 시) DB 활성 라인업 히트** — 스켈레톤을 DB 패턴으로 변환해 현재 유통 중인 `chip_prod_id` 조회
4. **검증 필요 항목** — 데이터시트/실측이 필요한 미확정 사항

"정확 카탈로그 매칭"이나 "근접 카탈로그 앵커"는 별도 섹션으로 분리하지 않는다. codebook 기반 스켈레톤 조립 과정에서 필요한 근거를 자연스럽게 언급하면 충분하다.

**스켈레톤 조립 과정 필수 포함**: position(1-7)별 코드 확정 근거, E-series 후보(비표준 명목 시), 사이즈/두께 필터 결과, 확정/미확정(`_` 또는 TBD) 구분, 후보 복수 시 우선순위 이유.

**DB 조회 단계 필수 포함**: 조회 패턴, `chip_prod_id` 목록/건수, 남은 모호성, 다건 시 포커스 후속 질문 1개.

**0건 조건 완화 단계 필수 포함**: 원본 패턴과 0건 확인, 고정 축/완화 방향 선택 질문, 재구성된 새 패턴.

## 실패 처리

근거가 불완전할 때는 아래 가드레일 문구를 명시적으로 사용한다.

- `정확한 카탈로그 일치 항목이 없어 후보 스켈레톤만 제공합니다.`
- `카탈로그 기반 사전 선정 결과이며 최종 주문 가능 P/N 확정이 아닙니다.`
- `정확한 꼬리코드 및 바이어스 유효용량은 데이터시트/실측 검증이 필요합니다.`
- `부분 패턴에 대한 현재 DB 히트는 제시했지만, 최종 1개 선택에는 추가 제약이 필요합니다.`
- `부분 패턴에 대해 현재 DB 히트가 없습니다. 변경 가능한 미해결 필드를 확인해 주세요.`
- `조건 완화 검색: [dimension]을(를) [old value] → [new value]로 변경하여 재검색합니다.`
- `[N]개 조건을 완화했으나 여전히 0건입니다. 와일드카드를 넓히거나 추가 조건 변경이 필요합니다.`

사용자가 카탈로그 근거를 초과하는 보증을 요구하면 보증은 거절하고, 가능한 최강의 카탈로그 기반 사전선정안을 제시한다.
